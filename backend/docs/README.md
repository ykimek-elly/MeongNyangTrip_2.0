# MeongNyangTrip Backend

반려동물 동반 장소 추천을 위한 Spring Boot 백엔드입니다.

이 프로젝트는 "장소를 많이 보여주는 서비스"보다, 사용자 상황에 맞지 않는 후보를 먼저 줄이고 그 근거를 설명 가능한 형태로 남기는 추천 흐름 설계에 집중했습니다. 사용자와 반려동물 정보, 현재 날씨, 위치 기반 장소 데이터, 정책/가이드 문서를 함께 사용하지만, 최종 추천 순위는 규칙 기반으로 계산하고 LLM은 그 결과를 설명하는 역할에 한정했습니다.

포트폴리오 관점에서는 AI 호출 자체보다 "추천 책임을 어디까지 모델에 맡기고 어디까지 서버가 통제하는가"를 보여주기 위한 구조입니다.

## 프로젝트 개요

- 목표: 반려동물과 외출할 때 조건에 맞는 장소를 추천
- 핵심 문제: 날씨, 거리, 장소 성격, 반려동물 특성을 한 번에 고려해야 함
- 설계 방향:
  - 추천 후보를 먼저 좁힌 뒤 점수화를 통해 일관된 기준으로 정렬
  - LLM이 순위를 직접 결정하지 않도록 서버 측에서 추천 근거를 통제
  - 외부 API와 생성형 AI 호출은 캐시로 감싸 비용과 응답 지연을 관리

## 왜 이렇게 설계했는가

단순 키워드 검색이나 거리순 정렬만으로는 "오늘 이 반려동물에게 적절한 장소인가"를 설명하기 어렵습니다. 반대로 LLM에게 전체 추천을 맡기면 결과 재현성과 통제가 약해집니다.

그래서 이 프로젝트는 추천을 두 단계로 분리했습니다.

1. 서버가 후보 장소를 필터링하고 점수화해 추천 순위를 결정
2. Gemini는 그 순위와 근거를 바탕으로 사용자에게 보여줄 추천 문장을 생성

이렇게 하면 추천 품질을 규칙으로 안정적으로 관리하면서도, 사용자에게는 자연어 설명을 제공할 수 있습니다.

## 추천 흐름

```text
사용자/대표 반려동물 조회
  -> 현재 위치 기준 날씨 조회 (Redis 캐시 우선)
  -> PostGIS 기반 반경 검색으로 가까운 거리의 후보 장소 수집
  -> 운영 여부 / 좌표 유효성 / 카테고리 / 날씨 정책으로 1차 필터링
  -> 반려동물 적합도 / 날씨 적합도 / 장소 환경 / 거리 / 보정 점수 계산을 통해 2차 필터링
  -> 상위 후보에 대한 설명 보강용 RAG 검색 (pgvector)
  -> 추천 근거를 포함한 프롬프트 생성
  -> Gemini 응답 생성 또는 Redis 캐시 재사용
  -> 추천 결과와 AI 응답 로그 저장
```

핵심은 "AI가 추천을 대신하는 흐름"이 아니라, "서버가 정한 추천 결과를 AI가 이해하기 쉽게 설명하는 흐름"입니다.

## 추천 로직 요약

### 1. 후보 추출

후보 추출 단계에서는 사용자의 현재 좌표와 날씨를 기준으로 너무 멀거나 현재 조건에 맞지 않는 장소를 먼저 제거합니다. 특히 날씨 상태에 따라 허용 반경과 실내/실외 조건을 달리 적용해, 점수 계산 전에 탐색 범위를 줄이도록 설계했습니다.

### 2. 점수 계산

최종 순위는 아래 요소를 합산해 계산합니다.

- 반려동물 적합도: 나이, 활동량, 크기, 성향, 선호 장소
- 날씨 적합도: 산책 지수, 강수, 기온, 바람
- 장소 환경: 반려동물 친화 정보, 안전/쾌적성, 기본 품질 지표
- 이동 편의성: 현재 위치와 장소 간 거리
- 보정 점수: 검증 여부, 설명/태그 충실도 등

점수 계산을 서버에 둔 이유는 추천 기준을 코드로 명시하고, 어떤 기준으로 추천했는지 로그로 남길 수 있기 때문입니다.

### 3. 설명 생성

상위 후보와 점수 근거를 바탕으로 Gemini가 미리 만든 프롬프트를 통해 최종 추천 문장을 생성합니다. 여기서 Gemini는 추천 순위를 결정하지 않고, 이미 계산된 결과를 사용자 친화적인 문장으로 풀어주는 역할만 수행합니다.

## RAG를 넣은 이유

RAG는 "AI를 더 똑똑하게 보이게 하기 위한 장치"가 아니라, 추천 설명에 필요한 배경 지식을 제한된 범위에서 보강하기 위한 수단으로 사용했습니다.

- 대상 문서: 반려동물 복지, 건강 관리, 공공장소 이용 정책 등 PDF 문서
- 저장 방식: PostgreSQL 기반 `pgvector`
- 사용 방식:
  - 반려동물 특성과 현재 날씨를 바탕으로 검색 질의 생성
  - 관련성이 낮은 문장은 키워드 기준으로 제외
  - 최종 프롬프트에는 짧게 압축한 컨텍스트만 포함

RAG는 추천 자체를 계산하지 않고, 추천 설명이 근거 없는 일반론으로 흐르지 않도록 보조합니다.

## 기술 선택과 이유

| 기술 | 선택 이유 |
| --- | --- |
| Spring Boot / JPA | 도메인별 계층을 분리하고 추천 파이프라인을 서비스 단위로 조합하기 쉬운 구조가 필요했습니다. |
| PostgreSQL | 사용자, 반려동물, 장소, 추천 로그를 한 DB에서 관리하기 위한 기본 저장소로 선택했습니다. |
| PostGIS | 반경 검색과 거리 계산을 애플리케이션 메모리에서 하지 않고 DB 레벨에서 처리해 후보 추출 비용을 줄이기 위해 사용했습니다. |
| Redis | 날씨 조회와 Gemini 응답처럼 반복 요청이 많은 구간의 지연과 비용을 줄이기 위해 TTL 캐시를 적용했습니다. |
| pgvector | 별도 벡터 DB를 추가하지 않고 기존 PostgreSQL 생태계 안에서 RAG 검색을 운영하기 위해 선택했습니다. |
| Gemini | 추천 순위를 맡기기보다, 서버가 계산한 결과를 자연어로 설명하는 생성 계층으로 사용했습니다. |
| Spring AI | Gemini 호출, 임베딩, 벡터 스토어 연결을 Spring 애플리케이션 안에서 일관되게 구성하기 위해 사용했습니다. |

## 캐시 전략

캐시는 성능 최적화보다 "외부 의존성의 불안정성을 흡수하는 완충층"이라는 관점으로 넣었습니다.

- 날씨 캐시: 격자 좌표 기준으로 조회 결과를 저장해 동일 지역 반복 호출 감소
- Gemini 캐시: 동일 프롬프트에 대한 응답 재사용으로 응답 시간과 호출 비용 감소

## 프로젝트 구조

```text
backend/src/main/java/com/team/meongnyang
├─ recommendation
│  ├─ service        # 후보 추출, 점수 계산, 프롬프트 생성, Gemini 연동
│  ├─ rag            # PDF 로딩, 벡터 검색, 컨텍스트 압축
│  ├─ cache          # 날씨 / Gemini 캐시
│  ├─ weather        # 날씨 조회 및 격자 변환
│  └─ log            # AI 응답 로그 저장
├─ place             # 장소 조회 및 검증
├─ user              # 사용자 / 반려동물 정보
├─ batch             # 장소 수집·보강 배치
└─ config            # Redis, AI, 공통 설정
```

## 현재 한계

- 추천 점수는 아직 규칙과 수동 가중치 중심이라, 실제 가중치 기준이 충분히 검증되지 않았습니다.
- 현재 추천 흐름은 사용자 위치를 실시간으로 받기보다 고정 좌표 기반으로 동작하는 부분이 있어 MVP 수준의 제약이 있습니다.
- RAG 문서군이 정책/가이드 중심이라 장소별 상세 맥락을 풍부하게 설명하는 데 한계가 있습니다.
- RAG를 Gemini의 근거 보강용으로 넣었지만, 실제 응답에서 이를 충분히 활용하지 않는 경우가 많아 유지 필요성을 다시 검토하고 있습니다.
- Gemini 응답은 캐시와 fallback을 갖고 있지만, 지연 시간 자체를 줄이는 최적화는 아직 진행 중입니다.
- 현재 지연 시간은 대략 RAG 2초, Gemini 4초 수준으로 두 구간만 합쳐도 약 6초가 소요됩니다.

## 개선 예정

- 추천 로그를 바탕으로 점수 가중치와 필터 기준 재조정
- 사용자 위치 입력과 이동 맥락을 반영한 추천으로 확장
- RAG 문서 선별 기준과 검색 품질 개선
- 캐시 hit ratio, 응답 시간, fallback 비율 등 운영 지표 측정
- 추천 결과에 대한 사용자 피드백 수집 구조 추가
