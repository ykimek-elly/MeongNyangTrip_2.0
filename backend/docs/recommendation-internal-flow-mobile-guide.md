# 추천 시스템 내부 흐름 모바일 가이드

## 1. 추천 시스템 목적

반려견과 현재 조건에 맞는 장소를 빠르게 1곳 추천하고, 그 이유를 짧게 설명해 알림까지 보내는 시스템이다.

---

## 2. 전체 흐름

1. 사용자와 날씨를 읽고 후보 장소를 모은다.  
2. 후보를 점수화해 1등 장소를 고른다.  
3. Gemini로 설명을 만들고, 필요하면 알림까지 보낸다.

핵심 클래스 흐름은  
`RecommendationController -> RecommendationQueryService -> RecommendationPipelineService` 이다.

---

## 3. 단계별 상세 설명

### 1단계. 요청 진입

`RecommendationController`가 `/api/v1/ai/walk-guide` 요청을 받는다.  
로그인 사용자 이메일을 꺼내 `RecommendationQueryService`로 넘긴다.

왜 하냐면  
추천 대상 사용자를 정확히 특정해야 하기 때문이다.

다음 단계는  
오늘 이미 보낸 추천이 있는지 확인하는 단계다.

### 2단계. 오늘 추천 캐시 확인

`RecommendationQueryService`는 `DailyRecommendationCacheService`를 먼저 본다.  
오늘 이미 보낸 기록이 있으면 오늘 결과를 그대로 돌려준다.

왜 하냐면  
같은 날 같은 사용자에게 매번 새 추천을 만들 필요가 없기 때문이다.

다음 단계는  
오늘 결과가 없을 때만 실제 추천 파이프라인으로 들어간다.

### 3단계. 사용자 중복 요청 잠금

`RecommendationPipelineService`는 `RecommendationDedupService`로 사용자별 lock을 건다.  
lock을 못 잡으면 추천 생성 중이라고 바로 응답한다.

왜 하냐면  
같은 사용자가 동시에 여러 번 눌렀을 때 AI 호출과 캐시 생성이 중복되기 때문이다.

다음 단계는  
lock을 잡은 경우에만 날씨와 후보 장소를 계산한다.

### 4단계. 사용자/반려견/날씨 읽기

파이프라인은 `RecommendationUserReader`, `RecommendationPetReader`로 대상 정보를 읽는다.  
사용자 좌표를 `WeatherGridConverter`로 바꿔 `WeatherCacheService`에서 날씨를 가져온다.

왜 하냐면  
추천은 현재 위치와 현재 날씨 영향을 크게 받기 때문이다.

다음 단계는  
현재 조건으로 갈 수 있는 후보 장소를 모은다.

### 5단계. 후보 장소 수집

`CandidatePlaceService.getInitialCandidates(...)`가 후보를 만든다.  
먼저 반경 내 장소를 가져오고, 필터를 단계적으로 적용한다.

왜 하냐면  
아무 장소나 점수 계산하면 비용이 크고, 결과도 이상해지기 때문이다.

다음 단계는  
후보가 있으면 결과 캐시 확인 후 점수 계산으로 간다.

### 6단계. 추천 결과 캐시 확인

`RecommendationContextKeyFactory`가 사용자, 반려견, 날씨 밴드, 후보 상위 목록, 최근 추천 장소로 key를 만든다.  
`RecommendationResultCacheService`에 동일 문맥 결과가 있으면 바로 반환한다.

왜 하냐면  
짧은 시간 안에 같은 조건이면 같은 결과를 재사용하는 게 빠르고 싸기 때문이다.

다음 단계는  
캐시가 없을 때만 점수 계산을 한다.

### 7단계. 점수 계산과 순위 결정

`PlaceScoringService.scorePlaces(...)`가 후보를 정렬한다.  
현재 실제 파이프라인은 `scorePlaceEntityDriven(...)` 기반 점수를 사용한다.

왜 하냐면  
후보 중에서 가장 신뢰 가능하고 반려동물 동반 정보가 좋은 장소를 골라야 하기 때문이다.

다음 단계는  
1등 장소와 상위 후보 근거를 모아 Gemini 입력을 만든다.

### 8단계. 설명용 근거 구성

`RecommendationEvidenceContextService`가 상위 3개 후보와 점수 근거를 문맥으로 묶는다.  
`RecommendationPromptService`가 이 문맥을 Gemini 프롬프트로 바꾼다.

왜 하냐면  
AI가 막연한 문장을 쓰지 않고, 이미 계산된 결과를 설명하도록 제한하기 위해서다.

다음 단계는  
Gemini 캐시를 먼저 보고, 없으면 실제 호출한다.

### 9단계. Gemini 설명 생성

`GeminiCacheService`는 문맥 캐시와 프롬프트 캐시를 둘 다 본다.  
없으면 `GeminiRecommendationService`가 Gemini를 호출한다.

왜 하냐면  
같은 추천이면 설명도 재사용할 수 있고, AI 비용과 지연을 줄일 수 있기 때문이다.

다음 단계는  
응답에서 알림용 한 줄 요약을 뽑아 최종 결과를 만든다.

### 10단계. 최종 결과 저장

파이프라인은 `RecommendationNotificationResult`를 만든다.  
추천 결과 캐시에 저장하고, 마지막 추천 장소 이력도 기록한다.

왜 하냐면  
같은 결과 재사용과 최근 추천 중복 방지를 같이 처리해야 하기 때문이다.

다음 단계는  
배치 알림인 경우 실제 발송 단계로 이어진다.

### 11단계. 알림 발송 연결

`NotificationScheduler -> NotificationBatchService`가 배치를 돌린다.  
대상 사용자 검사 후 같은 파이프라인으로 추천을 만들고 `NotificationService`로 보낸다.

왜 하냐면  
앱 안 조회용 추천과 알림용 추천이 같은 기준을 써야 결과가 어긋나지 않기 때문이다.

다음 단계는  
발송 성공 시 오늘 추천 캐시와 sent marker를 저장한다.

---

## 4. 후보 장소 수집 기준

기준 클래스는 `CandidatePlaceService`다.

반경 조회는 최대 `120개`까지 가져온다.  
최종 후보는 최대 `20개`만 남긴다.

추천 가능한 카테고리는  
`PLACE`, `DINING`, `STAY` 3개다.

거리 기준은 산책 등급별로 다르다.  
`GOOD 12km`, `CAUTION 8km`, `DANGEROUS 5km`다.

기본 필터는 아래 순서다.

1. 검증된 장소인지 본다.  
2. 폐업, 휴업, 운영 종료 태그가 있으면 뺀다.  
3. 좌표가 정상인지 본다.  
4. 추천 가능 카테고리인지 본다.  
5. 현재 거리 제한 안에 있는지 본다.  
6. 현재 날씨 정책에 맞는지 본다.

날씨 정책은 간단히 말하면  
위험할수록 실내 위주로 좁히는 구조다.

`DANGEROUS`면 실내만 통과한다.  
`CAUTION`이면 실내 또는 완전 야외가 아닌 곳을 본다.  
그 외는 비가 올 때 실내나 비야외 성격 장소를 우선 통과시킨다.

반려견 선호 장소(`preferredPlace`)가 있으면  
후보를 버리지는 않고 앞쪽으로 약하게 우선 배치한다.

---

## 5. 후보 완화 전략

이 부분이 면접에서 중요하다.

후보 수집은 한 번에 끝내지 않고  
`엄격 -> 완화 -> fallback` 3단계로 간다.

### 1차. 엄격 필터

날씨 조건까지 그대로 반영한다.  
예를 들어 위험 날씨면 실내만 남긴다.

왜 하냐면  
가능하면 현재 상황에 가장 안전한 후보를 먼저 쓰기 위해서다.

### 2차. 완화 필터

엄격 필터 결과가 없으면  
같은 기본 필터를 쓰되 날씨 조건만 완화한다.

왜 하냐면  
후보가 0개가 되면 추천 자체가 끊기기 때문이다.

### 3차. 핵심 fallback

그래도 없으면  
운영 여부, 좌표, 카테고리, 거리만 보고 후보를 만든다.

왜 하냐면  
최소한 갈 수 있는 장소 목록은 남겨서 서비스가 완전히 빈 결과가 되지 않게 하려는 것이다.

정리하면  
날씨 조건을 가장 먼저 느슨하게 풀고, 그래도 없으면 핵심 조건만 남긴다.

---

## 6. 점수 계산 기준

핵심 클래스는 `PlaceScoringService`다.

### 면접에서 꼭 말할 점

코드 안에는 예전 방식 점수 함수도 있다.  
하지만 현재 실제 파이프라인은 `scorePlaceEntityDriven(...)`를 사용한다.

즉, 최종 순위는 지금  
장소 메타데이터 품질 중심으로 계산된다.

### 실제 사용 중인 점수 구성

총점은 `100점 만점` 구조다.  
각 섹션 점수를 더하고 패널티를 뺀다.

섹션은 5개다.

1. `정책 적합도 30점`  
실내 동반 가능 여부, 정책 문구 명확성, 제한 조건, 카테고리 적합도를 본다.

2. `반려동물 편의시설 15점`  
전용 공간, 배변, 물그릇, 놀이터, 이동 편의, 청결/안전 신호를 본다.

3. `장소 신뢰도/품질 20점`  
공개 평점, 리뷰 수, AI 평점, 블로그 수를 합쳐 본다.

4. `블로그 감성 신호 15점`  
긍정 태그는 가점, 부정 태그는 감점한다.

5. `문맥 적합도 20점`  
카테고리, 태그, 숙소의 반려동물 운영 정보, 데이터 충실도를 본다.

### 패널티

패널티는 `장소 데이터 패널티 + 다양성 패널티`다.

장소 데이터 패널티는  
실내 불가, 야외만 가능, 크기 제한, 추가 요금, 사전 문의, 부정 태그가 많을 때 붙는다.

다양성 패널티는  
최근 추천한 장소가 또 1등 되는 걸 막기 위해 붙인다.

후보가 너무 적으면  
다양성 패널티를 약하게 하거나 아예 무시한다.

### 주의할 점

메서드 이름만 보면 사용자/반려견/날씨 맞춤 점수가 크게 들어갈 것 같지만,  
현재 실제 정렬은 장소 엔티티 정보 중심이다.

면접에서는  
"후보 단계에서 날씨와 거리로 먼저 걸러내고, 최종 점수는 장소 품질과 동반 가능 정보를 중심으로 계산한다"  
라고 말하면 안전하다.

## 7. 캐시 전략

캐시는 4개 층으로 보면 된다.

### 1. 날씨 캐시

`WeatherCacheService`가 격자 좌표 기준으로 날씨를 캐싱한다.  
안정 날씨는 `30분`, 심한 날씨는 `10분` TTL이다.

왜 쓰냐면  
같은 지역에서 기상 API를 계속 때리면 느리고 비싸기 때문이다.

### 2. 추천 결과 캐시

`RecommendationResultCacheService`는 같은 추천 문맥 결과를 `10분` 캐싱한다.

왜 쓰냐면  
같은 조건의 반복 요청에서 후보 계산과 AI 설명 생성을 다시 하지 않기 위해서다.

### 3. Gemini 캐시

`GeminiCacheService`는  
프롬프트 캐시와 문맥 캐시를 따로 가진다.

문맥 캐시는 `12시간`이다.  
프롬프트 캐시는 설정값 TTL을 쓴다.

왜 쓰냐면  
비슷한 상황이면 설명도 재사용 가능해서 AI 비용을 줄일 수 있기 때문이다.

### 4. 일일 추천 캐시

`DailyRecommendationCacheService`는  
그날 보낸 결과와 sent marker를 자정 이후까지 저장한다.

왜 쓰냐면  
오늘 이미 보낸 추천을 앱 조회와 알림에 똑같이 재사용하기 위해서다.

---

## 8. 중복 요청 방지 이유

`RecommendationDedupService`는 사용자별 lock을 `2분` 잡는다.

왜 하냐면  
동시에 여러 요청이 들어오면 날씨 조회, 점수 계산, Gemini 호출, 캐시 저장이 겹치기 때문이다.

결과적으로  
성능 낭비를 막고, 같은 순간 서로 다른 추천이 튀는 문제도 줄인다.

추가로 최근 추천 장소 이력도 `72시간` 정도 유지한다.  
이 이력은 같은 장소만 계속 추천되지 않게 하는 데 쓴다.

---

## 9. AI(Gemini) 역할

Gemini의 역할은  
`장소를 고르는 것`이 아니라 `선정된 결과를 설명하는 것`이다.

1등 장소는 이미 코드가 점수로 정한다.  
Gemini는 그 결과를 사람이 읽기 쉽게 풀어준다.

왜 설명 생성만 맡기냐면  
순위 결정까지 AI에 맡기면 결과가 흔들리고 검증이 어려워지기 때문이다.

현재 프롬프트도  
"순위는 서버가 이미 계산했으니 주어진 근거만으로 설명하라"는 식으로 제한한다.

또한 알림용 한 줄 요약은  
Gemini 응답의 `[알림요약]` 섹션 첫 줄만 뽑아 쓴다.

Gemini가 실패하면  
기본 fallback 문장을 사용한다.

---

## 10. 알림 연결 흐름

알림은 `NotificationScheduler`가 정해진 시간에 시작한다.  
그리고 `NotificationBatchService`가 실제 배치를 처리한다.

흐름은 이렇다.

1. 알림 대상 사용자 목록을 읽는다.  
2. 대표 반려견을 붙인다.  
3. `NotificationPolicyService`로 발송 가능 여부를 본다.  
4. 가능하면 추천 파이프라인으로 추천을 만든다.  
5. 1등 장소가 있으면 `NotificationService`로 보낸다.  
6. 성공하면 사용자 발송 시각, 일일 캐시, sent marker를 저장한다.

왜 이렇게 연결하냐면  
알림도 앱 조회와 같은 추천 기준을 써야 서비스가 일관되기 때문이다.

알림 정책에서 체크하는 것은  
활성 사용자, 일반 사용자 역할, 알림 허용, 전화번호 존재, 대표 반려견 존재, 오늘 미발송 여부다.

---

## 11. 실패 처리

실패 처리는 여러 단계에서 나뉜다.

### 추천 중복 실패

이미 같은 사용자 요청이 처리 중이면  
`RECOMMENDATION_IN_PROGRESS` 에러 코드로 바로 돌려준다.

### 후보 없음

후보가 비면  
장소 없이 메시지만 담은 empty recommendation을 반환한다.

### 전체 파이프라인 예외

파이프라인 전체가 예외면  
`RECOMMENDATION_FAILED` 코드의 error recommendation을 반환한다.

### Gemini 실패

타임아웃, 빈 응답, 실행 예외가 나면  
재시도 후 fallback 문장을 사용한다.

즉  
AI가 죽어도 추천 응답 자체는 끊지 않으려는 구조다.

### 알림 실패

배치에서는 실패 사유를 따로 모은다.  
예: 반려견 없음, 날씨 API 오류, 후보 없음, AI 오류, 알림 발송 실패.

왜 이렇게 나누냐면  
어디서 막혔는지 운영자가 바로 볼 수 있어야 하기 때문이다.

---

## ✔ 외워야 할 핵심 10개

1. 추천 목적은 반려견과 현재 조건에 맞는 장소 1곳을 고르고 이유를 설명하는 것이다.
2. 메인 흐름은 `후보 수집 -> 점수 계산 -> 설명 생성`이다.
3. 오늘 이미 보낸 추천은 `DailyRecommendationCacheService`에서 재사용한다.
4. 중복 요청은 사용자별 lock으로 막는다.
5. 후보 수집은 `엄격 -> 완화 -> fallback` 3단계다.
6. 거리 기준은 `GOOD 12km`, `CAUTION 8km`, `DANGEROUS 5km`다.
7. 실제 최종 점수는 현재 `scorePlaceEntityDriven` 기반이다.
8. 점수 핵심은 동반 정책, 편의시설, 신뢰도, 블로그 신호, 문맥 적합도다.
9. Gemini는 순위를 정하지 않고 설명만 만든다.
10. 알림은 같은 추천 파이프라인을 재사용하고, 성공 시 오늘 캐시를 저장한다.

---

## ✔ 면접에서 설명할 핵심 흐름

"사용자 요청이 오면 먼저 오늘 추천 캐시를 확인합니다.  
없으면 추천 파이프라인에 들어가서 사용자와 대표 반려견, 현재 날씨를 읽고, 날씨와 거리 기준으로 후보 장소를 모읍니다."

"후보는 엄격 필터, 완화 필터, fallback 순서로 줄입니다.  
그다음 장소의 반려동물 동반 정책, 편의시설, 평점과 리뷰, 블로그 신호, 데이터 품질을 점수화해서 1등 장소를 고릅니다."

"마지막으로 Gemini는 이미 정해진 1등 결과를 설명하는 역할만 합니다.  
알림 배치도 같은 파이프라인을 사용하고, 발송 성공 시 오늘 추천 캐시에 저장해서 앱 조회와 알림 결과를 맞춥니다."
