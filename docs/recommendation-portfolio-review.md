# recommendation 포트폴리오 면접 정리

## 1. 프로젝트 한줄 설명
반려동물 정보, 사용자 위치, 실시간 날씨, 장소 데이터를 조합해 산책/외출 장소를 추천하고 알림까지 보내는 백엔드 추천 시스템입니다.

## 2. 왜 만들었는지
- 반려동물 동반 장소는 많아 보여도 실제로는 날씨, 거리, 반려동물 성향까지 같이 고려해야 해서 사용자가 직접 고르기 어렵다고 봤습니다.
- 단순 장소 검색보다, "오늘 지금 이 사용자에게 맞는 곳 1곳을 바로 추천해주는 기능"을 만들고 싶었습니다.
- 면접 관점에서는 CRUD를 넘어서 추천 로직, 외부 API, 캐시, 배치, 알림까지 연결한 백엔드 흐름을 보여주기 위한 기능입니다.

## 3. 핵심 기능
- 사용자 인증 후 현재 사용자 기준 추천 조회 API 제공
- 기상청 날씨 API 조회 후 산책 가능 등급(`GOOD`, `CAUTION`, `DANGEROUS`) 계산
- 사용자 위치 주변 장소를 거리, 운영 여부, 카테고리, 날씨 조건으로 1차 필터링
- 반려동물 적합도, 날씨 적합도, 장소 환경, 거리, 최근 추천 이력 페널티를 반영해 점수화
- 상위 후보를 근거 데이터로 정리한 뒤 Gemini로 추천 설명 문장 생성
- Redis 기반 캐시로 날씨, Gemini 응답, 최종 추천 결과, 당일 발송 결과 재사용
- 중복 요청 락과 최근 추천 기록으로 동일 사용자 중복 추천 방지
- 배치로 추천 생성 후 카카오 알림 발송, 실패 사유 분류 및 로그 저장

## 4. 강점
- 추천 로직이 한 번에 끝나는 게 아니라 `날씨 -> 후보 추출 -> 점수화 -> AI 문장 생성 -> 캐시 -> 알림`으로 분리돼 있어 역할이 명확합니다.
- AI에 전부 맡기지 않고, 먼저 서버에서 점수와 근거를 계산한 뒤 AI는 설명 생성에만 쓰도록 설계했습니다.
- Redis 캐시를 여러 단계에 적용해서 비용과 응답 속도를 줄이려 한 점이 실무적입니다.
- 추천 결과를 로그로 남기고 최근 추천 장소에 페널티를 줘서 같은 장소만 반복 추천되는 문제를 줄였습니다.
- 배치 발송, 실패 사유 집계, 재사용 가능한 당일 캐시까지 있어 운영 관점까지 고려했습니다.

## 5. 약점
- 추천 품질이 키워드 기반 규칙과 수작업 점수 비중이 커서 데이터가 바뀌면 튜닝이 많이 필요합니다.
- `PlaceScoringService`처럼 책임이 큰 클래스는 유지보수 난도가 높아질 수 있습니다.
- 테스트가 주로 단위 테스트 중심이라 실제 Redis, 외부 API, 배치까지 묶은 통합 검증은 상대적으로 약할 수 있습니다.
- 장소 데이터 품질이 낮거나 태그가 부정확하면 추천 정확도도 같이 떨어집니다.

## 6. 개선 포인트
- 점수 기준을 설정 파일이나 정책 객체로 분리해서 운영 중 튜닝 가능하게 개선
- 추천 결과 클릭/저장/재방문 데이터를 쌓아 규칙 기반에서 점진적으로 데이터 기반 추천으로 확장
- 통합 테스트와 배치 시뮬레이션 테스트 추가
- 장소 메타데이터 정제 자동화로 태그 품질 개선
- 추천 사유를 사용자에게 더 짧고 명확하게 보여주는 응답 포맷 개선

## 7. 면접 예상 질문
- 왜 AI 추천인데 규칙 기반 점수화를 먼저 했나요?
- 추천 정확도를 어떻게 높이려고 했나요?
- 캐시는 왜 여러 단계로 나눴나요?
- 중복 추천은 어떻게 막았나요?
- 날씨 API나 AI 호출이 실패하면 어떻게 처리하나요?
- 이 기능에서 본인이 가장 신경 쓴 부분은 무엇인가요?

## 8. 답변 포인트
- AI만 쓰면 결과가 흔들릴 수 있어서, 먼저 서버에서 후보와 순위를 계산하고 AI는 설명 생성에만 사용했다고 말하면 됩니다.
- 정확도는 사용자/반려동물/날씨/거리/장소 정책/최근 추천 이력을 함께 반영한 점수화로 높이려 했다고 설명하면 됩니다.
- 캐시는 비용 절감, 응답 속도 개선, 당일 같은 추천 재사용을 위해 날씨/AI 응답/최종 결과/일일 발송 캐시로 나눴다고 답하면 됩니다.
- 중복 추천은 요청 락, 최근 추천 장소 기록, AI 로그 기반 페널티로 제어했다고 말하면 됩니다.
- 실패 시에는 fallback 응답, 에러 코드 반환, 배치 실패 사유 기록으로 서비스가 완전히 멈추지 않게 했다고 설명하면 됩니다.
- 가장 신경 쓴 부분은 "추천 품질"과 "운영 안정성"을 같이 잡으려 한 구조라고 정리하면 좋습니다.

## ✔ 1분 설명
이 프로젝트의 recommendation 기능은 반려동물 산책 또는 외출 장소를 사용자 상황에 맞게 추천하는 백엔드 시스템입니다. 사용자의 위치와 반려동물 정보, 실시간 날씨, 장소 데이터를 합쳐 후보를 먼저 고르고, 점수화해서 상위 장소를 정한 뒤 Gemini로 설명 문장을 만드는 구조입니다. 여기에 Redis 캐시를 넣어서 날씨 조회와 AI 호출 비용을 줄였고, 당일 추천 결과 재사용과 중복 요청 방지도 처리했습니다. 또 배치로 알림까지 보내도록 연결해서, 단순 추천 API가 아니라 실제 서비스 흐름까지 고려한 기능이라고 설명할 수 있습니다.

## ✔ 3분 설명
이 기능은 "오늘 이 사용자와 반려동물에게 어디를 추천할까"를 해결하기 위해 만들었습니다. 먼저 인증된 사용자를 기준으로 추천 요청이 들어오면, 위치를 기상청 격자로 바꿔 날씨를 조회하고 산책 가능 등급을 계산합니다. 그 다음 주변 장소를 가져와 운영 여부, 카테고리, 거리, 날씨 적합성으로 1차 필터링합니다.

그 뒤에는 `PlaceScoringService`에서 반려동물 적합도, 날씨 적합도, 장소 환경, 거리, 최근 추천 이력에 따른 페널티까지 반영해서 장소 점수를 계산합니다. 즉 AI가 장소를 직접 고르는 게 아니라, 서버가 먼저 근거 있는 순위를 만든 뒤 AI는 그 결과를 설명하는 역할만 하게 했습니다. 이렇게 한 이유는 추천 결과의 일관성과 제어 가능성을 높이기 위해서입니다.

운영 측면에서는 Redis 캐시를 여러 단계에 넣었습니다. 날씨 캐시, Gemini 응답 캐시, 최종 추천 결과 캐시, 당일 발송 캐시를 분리해서 비용과 응답 시간을 줄였습니다. 또 같은 사용자에게 동시에 요청이 몰리면 락으로 막고, 최근에 추천한 장소는 다시 너무 자주 나오지 않도록 페널티를 줬습니다. 마지막으로 배치에서 추천을 생성하고 카카오 알림을 발송하며, 실패 사유도 남기게 해서 실서비스형 구조를 만들려 했다고 설명하면 됩니다.

## ✔ 핵심 키워드
- 개인화 추천
- 규칙 기반 점수화
- 날씨 기반 필터링
- Redis 캐시
- 중복 요청 제어
- 추천 다양성 페널티
- Gemini 설명 생성
- 배치 알림 발송
- 운영 안정성
- 실서비스형 백엔드
